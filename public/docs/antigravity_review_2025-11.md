# Google Antigravity レビュー（2025年11月時点）

*最終更新: 2025-11-30*

---

## Antigravity とは？

Google Antigravity は、Google が 2025 年 11 月に公開した「エージェント・ファースト（agent‑first）」な AI 開発環境です。VS Code 系のエディタをベースにしつつ、Gemini 3 Pro などの大規模言語モデルが「エージェント」としてエディタ・ターミナル・ブラウザを直接操作し、設計・実装・検証までまとめて担当することを想定しています。

従来の「コード補完・一問一答チャット」型のアシスタントではなく、

- **Manager（マネージャ）ビュー**で「ミッション（タスク）」を定義し、
- エージェントが計画を立て、コードを書き、ブラウザでテストし、
- その過程を **Artifacts（アーティファクト）** と呼ばれる成果物（タスクリスト、実装計画、スクリーンショットやテスト結果レポートなど）として残す

というワークフローが特徴です。

本記事では、2025 年 11 月時点の情報をもとに、以下の観点から Antigravity をレビューします。

- コスト
- 利用 AI（どんなモデルを使えるか）
- 再学習について（どこまで「学習」してくれるのか）
- 仕様駆動開発（Spec‑Driven Development）との関係
- 始め方（インストールと最初の一歩）

## 1. コスト

### 1-1. 現時点の料金体系（プレビュー）

2025 年 11 月時点では、Antigravity は **「個人利用向けの無償パブリックプレビュー」** として提供されています。

- 個人向けプラン: 月額 **0 円**
- 対応アカウント: 個人の Google アカウント（Gmail）
- 対応 OS: Windows / macOS / 一部 Linux ディストリビューション
- 利用できるモデル: Gemini 3 Pro を中心に、Claude Sonnet 4.5、OpenAI GPT‑OSS など（後述）

ただし「完全に使い放題」ではなく、**Gemini 3 Pro などの呼び出しにはクォータ（レートリミット）が設定**されています。一般に言われているのは：

- 一定量使うと「クォータ上限」に達し、数時間（例: 5 時間）待つとリセット
- マルチエージェントで重いタスクを連発すると、意外とすぐに上限に達する

という挙動です。実験用・学習用としては十分ですが、「一日中がっつり業務で回し続ける」には物足りない場合があります。

### 1-2. 将来の有料化の可能性

公式にはまだ詳細が出ていないものの、

- Team / Enterprise プランの予告
- 競合ツール（Cursor / Windsurf / Codeium / Claude Code など）が概ね 月額 10〜30 ドル程度

であることから、**正式版では個人向けの有料プラン（月 10〜20 ドル程度）とチーム向けプラン**が登場するだろう、という見方が一般的です。

また、企業向けには

- Vertex AI 上の Gemini 3 Pro / Gemini 3 Deep Think などの API 課金（例: 入力 1M トークンあたり数ドル〜）

と組み合わせた形で、Antigravity を開発フロントエンドとして使う構成が想定されています。

> まとめると：  
> **今は「個人利用ならただし、レート制限あり」** という非常にお得な状態ですが、  
> 正式リリース時には、他社ツール並みのサブスクリプション＋トークン課金モデルになる可能性が高い、と考えておくのが無難です。

## 2. 利用 AI（どんなモデルが動いているか）

### 2-1. 中心となる Gemini 3 Pro

Antigravity の「標準エンジン」は、Google 最新の大規模言語モデル **Gemini 3 Pro** です。

Gemini 3 Pro は、先行モデル Gemini 2.5 Pro よりも

- 推論能力（複雑な仕様理解や長いコードベースの把握）
- マルチモーダル（テキスト・画像）の理解
- エージェントとしてのタスク分解・計画能力

が大きく向上したとされており、Antigravity のマルチエージェント機能を前提に設計されています。

Antigravity 上では、Gemini 3 Pro が

- エディタ上のコード
- ターミナル出力
- 組み込みブラウザで開いたドキュメントや Web ページ

などをまとめて読み込み、**「タスクの計画 → コード変更 → テスト → レポート作成」**までを自律的に行います。

### 2-2. 他社モデルとのマルチモデル対応

Antigravity は、Gemini 以外のモデルも利用できる「マルチモデル対応」をうたっています。

代表的には：

- **Anthropic Claude Sonnet 4.5**
- **OpenAI GPT‑OSS**（オープンソース系の GPT 系モデルを Google がホストしているもの）

などに対応しており、ワークスペースやタスクごとに利用モデルを切り替えることができます。  
「重い仕様策定は Gemini 3 Pro、軽い補完やリライトは軽量モデル」といった使い分けも想定されています。

### 2-3. エージェントの動ける範囲

Antigravity のエージェントは、単なるチャットボットではなく、

- エディタ（コードの閲覧・編集）
- ターミナル（コマンド実行）
- 組み込みブラウザ（ドキュメントやアプリの UI を表示）
- Git 操作（ブランチ作成・コミット・差分表示など）

に直接アクセスできます。

その結果、

- 新規プロジェクトのひな型作成
- 既存コードベースの大規模リファクタリング
- テストコード生成と実行
- ライブラリや API の調査と比較表の作成
- UI の実装と E2E テスト

など、「人間ならシニアエンジニアがやるようなフロー」をかなりの部分までエージェントに任せる設計になっています。

## 3. 再学習について（Google がデータを AI 改善に使うか？）

ここでは「再学習」を、  
**「Antigravity での操作やプロンプトなどが、Google 側の AI モデルや製品改善に使われるかどうか」**  
という意味で整理します。

### 3-1. Antigravity 無料プレビュー版のデータ利用

Antigravity を初回セットアップすると、「Antigravity – Terms of Use」という画面で

- Antigravity にはセキュリティ上の制限があること
- 機密性の高いデータには使いすぎない方がよいこと
- エージェントの動作は必ず人間が確認すべきであること

といった注意書きが表示されます。

そのうえで、同じ画面に **チェックボックス付きの同意項目** が出ます（要旨）：

> Google が、Antigravity 上での自分の操作データ（interactions data）を収集・利用することに同意する。  
> このデータは、Google / Alphabet の研究・製品・サービス・機械学習技術の評価・開発・改善のために使われる。  
> 同意は、アカウント設定やサポートへの連絡でいつでも取り消すことができる。

ここがポイントで、

- **チェックを入れると**  
  → Antigravity 上でのやり取りや操作ログが、  
    Google 側の「製品・サービス・機械学習技術の改善」のために利用されうる（≒ モデルの再学習や評価用データに回りうる）

- **チェックを外せば**  
  → Antigravity を利用するために必要な範囲（レスポンス生成、ログ保存、障害解析や不正利用検知など）には使われるものの、  
    「AI の改善用データセット」という意味では利用されない、という扱いになります。

また、セットアップ後の設定画面には

- `Enable telemetry` のような **テレメトリ用のトグル**

も用意されており、ここを OFF にすることで、同様の用途のデータ送信を抑制できる構成になっています。

### 3-2. 「AI の再学習」にどこまで使われると考えるべきか

同意文言では、「研究・製品・サービス・機械学習技術の改善」にデータを使う、とされています。

一般的に、Google を含む多くの AI プロバイダは

- **モデルの事後評価（どの回答が良かったかの統計分析）**
- **新しい学習用データの構築（良い／悪い応答例としての再利用）**

といった形で、ユーザーのインタラクションデータを「AI の改善」に用いています。

Antigravity の同意文言もその流れに沿ったものと考えられるため、

- **チェックを入れている場合**  
  → プロンプト、コード、エージェントの実行ログなどが、将来の Gemini 系モデルや関連 ML システムの  
    「学習・評価・改善」の材料として使われる可能性が高い

- **チェックを外している場合**  
  → サービス提供や安全性確保のための一時的な保存はあっても、  
    「モデル品質を上げるためのトレーニングデータ」には使われない

と理解しておくのが現実的です。

なお、どちらの場合も、

- 不正利用検知や障害調査、品質保証などの目的で  
  一定期間ログが保持される可能性があること
- 機密データを扱うときは、  
  「そもそも無料プレビュー版に投入してよいデータか？」を組織として判断する必要があること

には注意が必要です。

### 3-3. 企業向け（Gemini for Google Cloud 等）との違い

Antigravity の裏側では Gemini 3 Pro などのモデルが使われていますが、  
**「どの窓口から Gemini を呼び出すか」によって、データの扱いルールが変わる**のが Google の特徴です。

ざっくり整理すると：

- **Antigravity 無料プレビュー（個人アカウント）**  
  - 利用規約上、操作データを AI 改善に使うための同意項目が提示される  
  - チェックを外せば、改善用データとしての利用は拒否できる  
  - とはいえ「無料・プレビュー」という性質上、本番コードや極秘情報を入れるのは避けた方が無難

- **Gemini for Google Cloud / Gemini for Workspace などのエンタープライズ向けサービス**  
  - 公式ドキュメントでは、「プロンプトやレスポンスはモデルのトレーニングには使わない」ことが明記されている  
  - データ処理はデータ処理契約（DPA）や各種コンプライアンスの枠組みの中で行われる  
  - 一部、「フィードバックデータを共有すると製品改善に使う」オプション機能はあるが、その場合も  
    「どこまでが ‘モデル学習’ に含まれるか」は契約・ドキュメントに沿って個別に確認する必要がある

企業で Antigravity を本格的に試す場合は、

- 「Antigravity の UI から Gemini を呼んでいるのか」
- 「背後で利用しているのが、無料枠の Gemini なのか、  
   それとも Gemini for Google Cloud のような課金サービスなのか」
- 「自社が結んでいる契約上、データがモデル学習に使われるのか / 使われないのか」

を、情報システム部門やセキュリティ担当と一緒に確認しておくのが安全です。

### 3-4. 実務でのおすすめ設定と注意点

実際に Antigravity を使うときの「再学習まわり」の実務的なポイントをまとめると、次のようになります。

1. **セットアップ時の同意チェックをよく読む**  
   - 個人利用で「AI の改善に貢献してもいいし、学習データに使われても構わない」という場合は ON のままでもよいでしょう。  
   - 機密性の高いコードを扱う場合や、会社の方針が不明な場合は、一旦 OFF（同意しない）にしておくのが無難です。

2. **設定画面の Telemetry 関連のトグルを確認する**  
   - `Enable telemetry` などの項目がある場合は、  
     「どこまで送信されるのか」「チームとして許容できるか」を確認したうえで ON/OFF を決めます。

3. **Google アカウント全体の AI/アクティビティ設定も確認する**  
   - Gemini や他の Google AI サービスには、「履歴を学習に使うかどうか」を切り替えるアカウント設定があります。  
   - Antigravity 以外のプロダクトも含めて、「このアカウントのデータは学習に使ってよいか？」を一度見直しておくと安心です。

4. **本番コード・極秘情報は、無料プレビューには載せない**  
   - 規約上は拒否設定ができるとしても、  
     金融・医療・重要インフラなどのコードや顧客データは、  
     原則として「評価用の無料プレビューツール」ではなく、  
     エンタープライズ契約下の環境で扱う方が安全です。

5. **チームで使う場合は、セキュリティ／法務と相談して方針を決める**  
   - 「Antigravity はこの範囲の PoC まで」「本番系は Cloud 側の Gemini だけ」など、  
     ツールの使い分けルールを決めておくと、後からトラブルになりにくくなります。

---

このように、「再学習」という観点で見ると、Antigravity は

- **個人向け無料プレビューとしては、データを AI 改善に使うことを前提としつつ、ユーザーに明示的な同意・撤回の手段を用意しているツール**
- **企業利用では、Gemini for Google Cloud などの「プロンプトを学習に使わない」系サービスと組み合わせる前提で評価すべきツール**

という位置づけになります。

## 4. 仕様駆動開発（Spec‑Driven Development）との関係

### 4-1. SDD（Spec‑Driven Development）とは？

「仕様駆動開発（Spec‑Driven Development: SDD）」は、ざっくり言うと

1. まず **仕様（Spec）やストーリーを Markdown などで丁寧に書く**
2. 仕様をもとに AI エージェントに実装計画を立てさせる
3. 計画が固まってから初めてコードを書く（または書かせる）
4. 仕様と実装を常に同期させる（テストやドキュメントも含めて）

という開発スタイルです。

GitHub が提唱している「Markdown を仕様言語として扱う」発想や、各社の AI IDE が取り入れている「Plan モード」に近い考え方です。

### 4-2. Antigravity における SDD 的な設計

Antigravity はこの SDD の文脈をかなり強く取り入れており、公式・非公式のドキュメントでも、

- **実装前に Implementation Plan（実装計画）Artifact を出す**
- タスクを細かく分解した **Task List Artifact** を生成する
- 実装・テスト後には **Walkthrough / Test Report Artifact** を残す

といった「中間成果物（Artifacts）を第一級に扱う」設計が強調されています。

コミュニティのテンプレートでは、

- `AGENTS.md` にプロジェクト全体の役割分担やルールを書く
- `.antigravity/rules.md` に「シニアエンジニアとして振る舞う」「必ずテストを書く」などのプロンプトを定義
- `artifacts/plan_*.md` に実装計画を保存させる

といったパターンが紹介されています。

### 4-3. SDD 的ワークフローの具体例

Antigravity を使った SDD 風の一連の流れは、例えば次のようになります。

1. **仕様を書く**
   - `docs/specs/todo_app.md` のようなファイルに、機能要件・非機能要件・画面遷移などを Markdown で記述。
2. **ミッションを定義**
   - Manager ビューで「この仕様の TODO アプリを実装する」というミッションを作成し、関連ファイル（spec や既存コード）を添付。
3. **実装計画 Artifact を生成**
   - エージェントに Implementation Plan Artifact を作らせ、  
     アーキテクチャ、テーブル設計、ルーティング構成、テスト方針などをレビュー。
4. **計画をレビュー＆修正**
   - Plan Artifact にコメントでフィードバックし、必要に応じて再生成させる。
5. **コード実装 & テスト**
   - 計画承認後、エージェントにコードを書かせる。  
     同時にテストコードや簡易ドキュメントも生成させる。
6. **Walkthrough / Test Report を確認**
   - 最後に Walkthrough Artifact で、どのようなテストを実行し、何が確認できているかをチェック。
7. **人間が最終レビュー**
   - 重要な箇所は必ず人間が差分レビュー・動作確認を行う。

このように、Antigravity は

> 「コードを書く IDE」というより、  
> **「仕様・計画・証拠（Artifacts）を中心に、エージェントをオーケストレーションする環境」**

として設計されていると言えます。

### 4-4. SDD 的に見たメリットと注意点

**メリット**

- 大規模変更でも、**「何をするのか」→「どうやるのか」→「実際の変更」→「検証結果」**が文書として残る
- チーム開発でレビューしやすい（Artifacts がそのままレビュー対象になる）
- 要件の抜け漏れや、「気づいたら別のものができていた」問題を減らせる
- AI エージェントの暴走を「仕様レベル」でコントロールしやすい

**注意点**

- 小さなタスクでもいちいち Artifact を作ると、逆に手間が増えることもある
- 仕様やルールのメンテナンスをサボると、かえって混乱しやすい
- プレビュー版ゆえの不安定さ（バグ・クォータ制限・レスポンスのムラ）もあり、  
  現時点では「完全自動開発」の幻想は持たない方がよい

## 5. 始め方（インストール〜最初のミッションまで）

ここでは、一般的な開発者が Antigravity を試してみる際の「最小限の手順」をまとめます。

### 5-1. 前提条件

- 個人の Google アカウント（Gmail）
- インターネット接続
- 対応 OS の PC
  - Windows 10 / 11
  - macOS（Apple Silicon / Intel）
  - 一部 Linux ディストリビューション
- ブラウザとして Google Chrome が推奨

ローカルでモデルを動かすわけではないため、GPU は必須ではありませんが、  
Antigravity 自体は重めの IDE なので、**メモリ 16GB 以上**くらいあると快適です。

### 5-2. インストール手順（概要）

1. **公式サイトへアクセス**
   - ブラウザで `https://antigravity.google` もしくはダウンロードページを開きます。
2. **インストーラをダウンロード**
   - 自分の OS（Windows / macOS / Linux）に対応するインストーラをダウンロード。
3. **インストール**
   - OS ごとのウィザードに従ってインストールします。
4. **初回起動＆ Google アカウントでログイン**
   - 起動すると Google ログイン画面が表示されるので、個人の Google アカウントでログインします。
5. **必要な権限を確認**
   - ファイルアクセスやブラウザ制御などの権限要求が表示されるので、内容を確認して許可します。
   - 機密性の高いコードや秘密情報を扱う場合は、この権限が自組織のポリシーに適合するか事前に確認してください。

### 5-3. 最初のワークスペースを作る

1. **既存リポジトリを開く or 新規プロジェクトを作る**
   - 「Open Folder」から既存の Git リポジトリを開くか、サンプルプロジェクトを作成します。
2. **Manager ビューを開く**
   - サイドバーから Manager（もしくは Missions）ビューを開きます。
3. **最初のミッションを作成**
   - 例:「既存 TODO アプリにダークテーマを追加する」など、シンプルなタスクをミッションとして登録します。
4. **Implementation Plan Artifact を生成**
   - エージェントに「まず実装計画を出して」と指示し、計画の内容をレビューします。
5. **計画にコメントでフィードバック**
   - 不足している観点（アクセシビリティ、テスト戦略など）があれば、Artifact にコメントして修正させます。
6. **実装を実行させる**
   - 計画が納得できる状態になったら、エージェントに実装・テストを進めさせます。
7. **Walkthrough / テスト結果を確認**
   - 最後に Walkthrough Artifact を読み、必要であればエージェントに追加の修正を依頼します。

### 5-4. セキュリティ上の注意（プレビュー版ならでは）

リリース直後から、Antigravity のエージェント権限を悪用する攻撃手法や、設定ファイルの扱いをめぐる脆弱性の指摘も出ています。特に、

- `.env` などに保存された秘密情報（API キー、データベース認証情報など）
- 社外秘のソースコードや設計書

を扱う際には、現時点のプレビュー版を **本番環境と同じ感覚で使うべきではありません。**

実際に業務で試す場合は、

- テスト用のサンドボックス環境を用意する
- 機密情報を含まないダミーデータで検証する
- エージェントに渡すファイルやフォルダを限定する

といった対策をとるのが現実的です。

## 6. まとめ：どんな人に向いているか？

### 6-1. 向いているケース

- AI コーディングツールの次のステップ（エージェント / SDD）を体験したい人
- 個人開発やサイドプロジェクトで、設計〜実装〜テストを AI と一緒に回してみたい人
- 仕様駆動開発や Artifact ベースのレビューに興味があるチーム
- Gemini 3 Pro を試してみたいが、API 連携や Vertex AI のセットアップは面倒だと感じている人

### 6-2. まだ慎重に検討した方がよいケース

- 機密情報を多く含むプロダクションコードを日常的に扱うチーム
- 強いセキュリティ要件（規制産業・金融・医療など）があり、ツールの評価に時間をかける必要がある組織
- 既に Cursor や GitHub Copilot などに十分投資しており、短期的にツールを乗り換える予定がないチーム

現時点の Antigravity は、

> 「未来の AI 開発フローを体験できる実験的な IDE」  

という位置づけで捉えるのがバランスが良いでしょう。  
本番運用を全面的に置き換えるというよりは、

- PoC / プロトタイピング
- 新しい開発スタイル（SDD / エージェント協調）の検証
- 個人開発・小規模チームでのトライアル

から試していくのがおすすめです。

## 用語ミニ解説

- **LLM（Large Language Model / 大規模言語モデル）**  
  大量のテキストデータを学習し、人間のような文章生成や要約、推論を行える AI モデルの総称。Gemini 3、GPT 系、Claude などが代表例。

- **エージェント（Agent）**  
  LLM に「ツール実行権限」や「目標達成のための自律行動」を持たせたもの。  
  Antigravity では、エディタ・ターミナル・ブラウザなどを操作する開発用エージェントが中心。

- **Artifacts（アーティファクト）**  
  Antigravity のエージェントが作る「検証しやすい成果物」。  
  例: タスクリスト、実装計画、テストレポート、スクリーンショット、ブラウザ録画など。  
  生のコードやログよりも、レビューしやすい形にまとめたもの。

- **仕様駆動開発（Spec‑Driven Development, SDD）**  
  先に仕様やストーリーを Markdown などで書き、それをもとに AI エージェントが計画・実装・テストを行う開発スタイル。  
  Antigravity では、Implementation Plan Artifact や AGENTS.md / rules.md などを通じて、この考え方が強く取り入れられている。

- **再学習 / ファインチューニング（Fine‑Tuning）**  
  既存の LLM に対して、特定のデータセット（自社のドキュメントやコードなど）で追加学習を行い、特定用途向けに性能を最適化すること。  
  Antigravity の一般ユーザー向けプレビューでは、LLM 本体のファインチューニングを直接操作する機能は提供されておらず、主にワークスペース内のナレッジベースとルール設計で「学習させる」形になっている。
